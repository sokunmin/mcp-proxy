## Project Overview and Agent Context

This document provides a comprehensive overview of the `mcp-proxy` project, its modifications, the challenges faced, and the solutions implemented. It is intended to bring any AI agent up to speed on the full context of the repository.

### 1. Initial Goal: A Multi-Server MCP Proxy

The base project, `mcp-proxy`, is a utility to proxy requests between a stdio-based client (like Claude Desktop) and a server using SSE (Server-Sent Events) or StreamableHTTP. The user's objective was to extend this functionality to run **multiple, different MCP servers** behind a single `mcp-proxy` instance. This would allow a single endpoint to provide access to a variety of tools (e.g., `fetch`, `time`, `github`, etc.).

To achieve this, the user made several key modifications:

- **`servers.json`**: This is the core configuration file created by the user. It contains a JSON object defining all the named servers that the proxy should manage. Each entry specifies the server's name and the command needed to run it (e.g., `"command": "uvx", "args": ["mcp-server-fetch"]`).

- **`install_mcp_packages.py`**: A custom Python script written by the user. Its purpose is to be run during the Docker build process. It parses `servers.json` and pre-installs all the required `uvx` (Python) and `npx` (Node.js) packages into the Docker image. This is a critical optimization to avoid slow, on-demand package installation when the server is running.

- **`Dockerfile`**: The user heavily modified the Dockerfile into a multi-stage build to create a self-contained, efficient image.
    - **Build Stage (`uv`)**: This stage installs the `mcp-proxy` application and its Python dependencies. Crucially, it then runs the `install_mcp_packages.py` script to pre-install all the tools from `servers.json`.
    - **Final Stage**: This stage creates a clean production image. It copies the Python virtual environment (`.venv`) and the pre-installed `uvx` and `npx` packages from the build stage. This ensures the final image has all dependencies and tools baked in without needing the build tools themselves.

- **`docker-compose.yml`**: Created by the user for easy local testing of the containerized application. It demonstrates how to run the service, map the port, and pass the necessary command-line arguments to the `mcp-proxy` entrypoint, specifically `--named-server-config ./servers.json`.

- **`smithery.yaml`**: This was updated for deployment on the Smithery platform. The `runtime` was set to `container`, and the `commandFunction` was modified to correctly start the proxy inside the container, telling it to listen on `0.0.0.0` and find the configuration at `/app/servers.json`.

### 2. The Problem: Deployment Timeout

When the user deployed this setup to Smithery, the deployment failed with a critical error:

```
Failed to scan tools list from server: McpError: MCP error -32001: Request timed out
```

**Root Cause Analysis:** The problem was **eager loading**. The original logic in `src/mcp_proxy/mcp_server.py` would read the `servers.json` file at startup and immediately loop through every single server defined within it. For each server, it would spawn a new subprocess (e.g., `uvx mcp-server-fetch`, `npx @modelcontextprotocol/server-github`, etc.).

The combined time required to start all of these subprocesses was significant. Smithery's deployment process includes a health and tool-scanning check that has a specific timeout. Because the `mcp-proxy` was busy launching all its child processes, it could not respond to Smithery's check in time, leading to the timeout and a failed deployment.

### 3. The Solution: Lazy Loading

The solution was to refactor the server to implement **lazy loading**. The proxy should start instantly and only launch a tool's subprocess when the first request for that specific tool arrives.

This was achieved by making the following changes to `src/mcp_proxy/mcp_server.py`:

1.  **`asyncio` Import**: `import asyncio` was added to support the asynchronous locking mechanism.

2.  **`LazyServerProvider` Class**: This new class is the heart of the solution. It was designed to manage the on-demand startup of server instances.
    -   **`__init__`**: It initializes with the server configurations and an `AsyncExitStack` but does *not* start any processes. It creates a dictionary of `asyncio.Lock` objects, one for each named server, which is crucial for preventing race conditions.
    -   **`get_app` method**: This is the core logic. When called for a server name:
        1.  It first checks if an application instance for that name is already cached. If so, it returns it immediately.
        2.  If not, it acquires the `asyncio.Lock` specific to that server's name. This ensures that even if 100 requests arrive for the same server at once, the startup process is only initiated a single time.
        3.  After acquiring the lock, it double-checks if the app was created by another request while it was waiting.
        4.  If the app still doesn't exist, it then proceeds to start the subprocess for that *one* server using `stdio_client`.
        5.  Once the server is running, it caches the Starlette application object and returns it.

3.  **Refactored `run_mcp_server` Function**:
    -   **Before**: The code looped through `named_server_params.items()` and started each server inside the loop, mounting the routes statically.
    -   **After**: The loop was replaced. It now instantiates the `LazyServerProvider`. The new loop still iterates over the server names but does not start them. Instead, for each server, it creates a small, dynamic `app_factory` function. This factory is what gets passed to Starlette's `Mount`. When a request comes in (e.g., for `/servers/fetch/sse`), Starlette calls the factory, which in turn calls `lazy_server_provider.get_app('fetch')`, triggering the on-demand startup.
    -   The global status is also updated more accurately. At startup, servers are marked as `"available"`. Their status only changes to `"running"` or `"failed"` after the first attempt to access them.

By implementing this solution, the `mcp-proxy` server's initial startup time became nearly instantaneous, as it no longer had to wait for any child processes. This allowed it to pass the Smithery health check, resolving the deployment timeout error.
